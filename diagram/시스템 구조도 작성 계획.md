# 시스템 구조도 작성 계획

## 목표

RC_Design 프로그램의 **전체 계산 흐름을 완전히 이해**할 수 있는 시스템 구조도를 단계적으로 구축한다.
최종 목표는 각 파일의 함수 호출 관계, 계산식(KDS 공식), 변수 흐름까지 모두 포함하는 **극단적으로 디테일한 다이어그램**이다.

## 현재 상태

- `diagram.html` — **Cytoscape.js** 기반 (Mermaid에서 전환)
- 우클릭 접기/펼치기, 휠 줌, 드래그 이동, 위치 자동 저장(localStorage)
- 겹침 방지 로직 (같은 부모의 형제 노드끼리 자동 밀어냄)
- 0~2단계 완료: 파일 구조 + 데이터 흐름 + calc_manager 내부 상세

## 작성 원칙

- **바깥→안쪽**: 전체 데이터 흐름 먼저, 개별 엔진 내부는 나중에
- **큰 흐름→세부 계산**: 함수 호출 관계 먼저, 계산식은 그 다음
- **2패스 방식**: 각 엔진마다 ①함수 트리 → ②계산식 순서로 작업
- Cytoscape.js compound 노드로 기존 다이어그램에 점진적으로 살을 붙임

---

## 단계별 작업 계획

### 1단계: 데이터 흐름도 (inputs → engines → results)

**왜 먼저?** 모든 연결의 뼈대. 이걸 모르면 나머지가 공중에 뜸.

**작업 내용:**
- `ui/input.py`의 `render_input_section()`이 반환하는 **inputs dict의 전체 키 목록** 정리
- 각 키가 어느 엔진의 어떤 파라미터로 전달되는지 매핑
  - 예: `inputs['L_x']` → `BeamAnalyzer.__init__(L=...)` → `results['beam_x']['h']`
- `calculation_manager.py`의 `perform_calculations()`이 반환하는 **results dict의 전체 키 구조** 정리
  - 최상위 키: `beam_x`, `beam_y`, `columns`, `slab`, `loads`, `ground_beam` 등
  - 각 하위 키의 의미와 단위
- 다이어그램에 추가: inputs dict → calc_manager → results dict 흐름을 키 이름과 함께 표시

**완료 기준:** "이 입력값이 어디로 가서 어떤 이름으로 돌아오는지" 다이어그램만 보고 추적 가능

---

### 2단계: calculation_manager 내부 구조

**왜 두 번째?** 전체 계산의 지휘자. 엔진을 보기 전에 "누가 뭘 호출하는지"부터 알아야 함.

**작업 내용:**
- `perform_calculations()` 함수의 실행 순서 (실제 코드 기준)
  1. 슬래브 두께 산정 + 하중분배 (1방향/2방향 판별)
  2. 슬래브 설계 → _design_slab()
  3. X방향 보 → _design_beam_with_deflection() (수렴루프)
  4. Y방향 보 → _design_beam_with_deflection() (수렴루프)
  5. 하중조합 생성 → _build_column_load_combos() (1.4D, 1.2D+1.6L, 지진 시 추가)
  6. 지배조합 선택 (P-M severity 기준)
  7. 기둥 설계 → _design_column_with_slenderness() (세장비+자중 수렴)
  8. 접합부 전단 + IMF 상세 (해당 시)
  9. 바닥보 설계 (해당 시)
  10. 결과 조립 → results dict 반환
- 하중조합 공식 목록 (KDS 조항 번호 포함)
- 기둥 지배 조합 선정 로직 (P-M severity 기준)
- 다이어그램에 추가: calc_manager 블록을 확장하여 내부 순서도 표시

**완료 기준:** calc_manager가 어떤 순서로 뭘 호출하고, 하중조합이 어떻게 적용되는지 파악 가능

---

### 3단계: beam_engine 함수 호출 트리 (계산식 제외)

**왜 세 번째?** 가장 복잡한 엔진. 함수 관계만 먼저 파악하고, 수식은 4단계에서.

**작업 내용:**
- `BeamAnalyzer` 클래스의 전체 메서드 목록
- `run()` → 내부에서 호출하는 함수 순서
  - `_flexural_design()` → `_calc_required_As()` → ...
  - `_shear_design()` → `_calc_Vc()` → `_calc_stirrup_spacing()` → ...
  - `_deflection_check()` → `_calc_Ie()` → ...
- **수렴루프 구조** (중요!)
  - 강도부족 → h+50 or 2단배근
  - 폭부족 → b+50
  - 처짐불만족 → h+50
  - 루프 종료 조건
- 각 함수의 입력 파라미터와 반환값 (타입/단위만, 수식은 아직)
- 다이어그램에 추가: beam_engine subgraph 안에 함수 호출 트리 + 수렴루프 플로우차트

**완료 기준:** beam_engine의 어떤 함수가 어떤 함수를 부르는지, 수렴루프가 어떻게 도는지 파악 가능

---

### 4단계: beam_engine 계산식 (KDS 공식)

**왜 네 번째?** 3단계의 함수 안에 수식을 채워넣는 작업.

**작업 내용:**
- 각 함수 안의 핵심 계산식을 KDS 조항 번호와 함께 추가
  - 휨: `a = As·fy / (0.85·fck·b)`, `φMn = φ·As·fy·(d-a/2)` — KDS 41 20 22 4.2
  - 전단: `Vc = (1/6)·√fck·b·d` — KDS 41 20 22 4.3
  - 처짐: `Ie = Icr + (Ig-Icr)·(Mcr/Ma)³` — KDS 41 20 22 4.4
  - 최소/최대 철근비: `ρ_min`, `ρ_max` — KDS 41 20 22 4.2.3
- 단위 변환 지점 표시 (kN·m → N·mm 등)
- 다이어그램에 추가: 각 함수 블록 안에 수식 텍스트 삽입 (또는 클릭 시 펼침)

**완료 기준:** 각 함수가 어떤 KDS 공식을 쓰는지, 단위 변환이 어디서 일어나는지 파악 가능

---

### 5단계: column_engine 함수 + 계산식

**작업 내용:**
- `ColumnAnalyzer` 클래스 메서드 목록 + 호출 트리
- 핵심 로직:
  - P-M 3점 포락선 (A점: 순압축, B점: 균형, C점: 순휨)
  - B-C 구간 50분할 이분법 (60회 반복)
  - Bresler 쌍축휨: `1/φPn = 1/φPnx + 1/φPny - 1/φPn0`
  - 세장비: `λ = k·lu/r`, 모멘트 확대계수 `δns`
  - EI 공식: `EI = (0.2·Ec·Ig + Es·Ise) / (1+βdns)`
- φ 전이구간 로직 (현재 미반영 — 감사항목 ①번)
- 다이어그램에 추가: column_engine subgraph

**완료 기준:** P-M 다이어그램 산정 과정, Bresler 방법, 세장비 계산 흐름 파악 가능

---

### 6단계: slab_engine 함수 + 계산식

**작업 내용:**
- `SlabAnalyzer` 클래스 — 보보다 단순 (1방향 고정단 1m 스트립)
- 휨/전단/처짐/배력근 계산 흐름
- 보와의 차이점 명시 (수렴루프 없음, 스트립 폭 1m 고정 등)
- 다이어그램에 추가: slab_engine subgraph (비교적 간단)

**완료 기준:** 슬래브 설계가 보와 어떻게 다른지, 어떤 공식을 쓰는지 파악 가능

---

### 7단계: UI ↔ 엔진 연동 (오버라이드 시스템)

**왜 이 시점?** 계산을 다 이해한 후 "사용자 조작이 어떻게 계산에 반영되는지" 파악.

**작업 내용:**
- `session_state` 키 목록과 역할
- 오버라이드 시스템 흐름:
  1. 사용자가 output 탭에서 단면 크기 수정
  2. `_on_size_change()` 콜백 → `modified_sizes` set에 추가
  3. 다음 계산 시 `_pick_size()`가 override 값 사용
  4. 수렴루프는 override된 값을 초기값으로 시작
- PDF 파싱 → session_state → input 반영 흐름
- 다이어그램에 추가: UI 상호작용 플로우

**완료 기준:** 사용자 입력/수정이 계산에 어떤 경로로 반영되는지 파악 가능

---

### 8단계: 시각화 데이터 흐름

**왜 마지막?** 출력단은 계산을 다 이해한 후에 봐야 의미 있음.

**작업 내용:**
- results dict의 어떤 키가 어떤 plot 함수에 전달되는지
- 각 시각화 함수의 파라미터 매핑
  - `plot_rebar_section(h, b, As_top, As_bot, ...)` ← `results['beam_x']`의 어떤 키
  - `plot_pm_diagram(points, ...)` ← `results['columns'][i]`의 어떤 키
- 3D 프레임의 데이터 조립 과정 (frame_3d.py의 복잡한 매핑)
- 다이어그램에 추가: visualization subgraph 상세화

**완료 기준:** 계산 결과가 어떤 그래프의 어떤 부분에 표시되는지 파악 가능

---

## 작업 방식

- 각 단계마다 `diagram.html`의 Cytoscape.js 노드/엣지를 추가
- 기존 파일 레벨 다이어그램은 유지하면서 **compound 노드 내부를 확장**
- 접기/펼치기로 복잡도 관리 가능
- 각 단계 완료 시 이 파일의 해당 섹션에 ✅ 표시
